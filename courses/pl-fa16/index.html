Programming Languages (Fall 2016)

ENGR 3599A / Tue 16:00-18:30 / AC 326

This course is an introduction to the theory, design, and implementation of programming languages. Topics covered may include: semantics of programming languages, types, higher-order functions and lambda calculus, objects, laziness, continuations, monads, objects, and concurrency. Class work revolves around the implementation of interpreters for a variety of small programming languages.

Course Information

Prerequisites: ENGR 2510

Instructor: Riccardo Pucella (send email)

Textbook: We don't have a textbook. There is nothing out there that captures quite what I want to do in this course. I will find readings for us to supplement the lectures. If you're looking for reference books that cover some of the topics we'll cover, the following should help:

  • Glynn Winskell, Formal Semantics of Programming Languages. This book specifically covers formal semantics, and not implementation. There are many excellent textbooks on semantics that have been published in the last few years. This one is a bit older, but it's a classic, and you should be able to find good used copies.
  • Shriram Krishnamurthy, Programming Languages: Application and Interpretation. This is the closest book in spirit to what I want to cover in the course. It is freely available online. It uses Racket, a dialect of Scheme. Definitely worth a read.

Programming: All programming in the course will be done in Python (version 2.7). I will assume working knowledge of the language. We will largely follow the PEP8 style guide, and you may also want to read through this as well.

All work to be done in teams of two. Late homeworks will be penalized (10% per 24 hours) and no homework will be accepted after its solution has been discussed in class.

  • Sep 27:Cancelled

    Homework 4, due next Monday.

    The first question of the homework involves front-end transformations. Basically, we can add surface syntax expressions that does not require an associated abstract representation, if we transform what is expressed in the surface syntax into simpler expressions for which we do have abstract representations. How do those kind of transformations relate to syntax extensions (or macros) as implemented by Scheme or by Rust? The short answer: those kind of syntax extensions are exactly the sort of transformations that we can implement by hacking the parser in the way I described. What those language do is that they allow transformations to be expressed within the language, and their parser can look up those transformations while parsing, instead of requiring the transformations to be hard-wired into the parser. Same process, slightly more flexible approach. (There's a cost to that approach: the parser needs to be more complex because it needs to look up applicable transformations at suitable points. It might be a very nice project to figure out ways of doing just that.)

  • Nov 15:Types

    Notes from the lecture.

    Sample code from the lecture, with a simple type checker.

    Wikipedia's entry on type systems is reasonable.

    At the end of lecture, I mentioned the problem of giving a reasonable type to reference cells. The sample code types function ref as (-> ((int)) (ref int)), and therefore will only work to create integer reference cells. There is no way, say, to create a Boolean referenece cells. A more general type for reference cells uses type variables to give a generic type to ref, along the lines of (-> ((a)) (ref a)). The modifications necessary to the type checker are not extensive, except for the case of ECall, which requires unification in order to figure out the types to assign to the type variables to obtain the result type of the call. (I may provide some sample code in the future for this, or it can be a final project.) Type systems with generics are known as polymorphic type systems.

    Adding generics are one way to extend type systems to type more expressions while still maintaining safety. More advanced type systems using dependent types are even more expressive, and also more complex to check.

  • Dec 6:Program verification

    A pretty thorough introduction to Hoare Logic by Mike Gordon.

    ESC/Java, which extends type extending for Java with a form of pre- and post-conditions checking inspired by Hoare Logic.

    Here is the encoding for lists using functions that I showed in class: 

      (defun cons (x xs) (function (f g) (g x xs)))

  (defun nil () (function (f g) (f)))

  (defun nil? (lst) (lst (function () true) (function (x xs) false)))

  (defun head (lst) (lst (function () false) (function (x xs) x)))

  (defun tail (lst) (lst (function () false) (function (x xs) xs)))
  
  (defun length (lst) (if (nil? lst) 0 (+ 1 (length (tail lst)))))
