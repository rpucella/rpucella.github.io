Programming Languages (Spring 2020)

ENGR 3599 / Time 1:30-4:00pm / AC 304

This course is an introduction to the theory, design, and implementation of programming languages. Topics covered may include: semantics of programming languages, types, higher-order functions and lambda calculus, objects, laziness, continuations, monads, objects, and concurrency. Class work revolves around the implementation of interpreters for a variety of small programming languages.

  • Feb 6:Functions

    Slides from the lecture.

    Sample code from the lecture: functions with global definitions, functions as values, functions with static binding, and functions with dynamic binding,

    Programming languages that take expressions and functions as their core (if not sole) programs are usually called functional programming languages. A good overview of some advantages of the functional programming approach is given by John Hughes' Why Functional Programming Matters. The first three sections at least should be pretty accessible.

    A very short description of the difference between static binding and dynamic binding (here, and in many places, called static/lexical scoping and dynamic scoping)

    Here is the example I used in class that illustrates that Python uses static binding (like all modern languages): 

      def test (x):

    def foo (y) :
      return x + y

    def bar (x):
      return foo(x)

    return bar(200)
    Evaluating test(10) yields 210, which is what static binding tells you should be the answer. If Python used dynamic binding, the result would be 400, since the binding of x to 200 in function bar would be used to resolve the x in foo when evaluating bar(200).

    If you're into it, you can read about the upward funarg problem that motivated the invention of closures. Be warned, though: it gets deep in the guts of language implementation using a stack à la C.

  • Feb 20:Parser Transformations

    Sample code from the lecture.

    The idea that we can keep the abstract representation simple while still allowing an extensive surface syntax by using the "trick" of parsing some of the surface syntax into a complex abstract representation is widespread. For instance, the surface syntax (and x y z) can parse into roughly EIf(x,EIf(y,z,false),false). In other words, we don't need to introduce abstract representation for everything in the surface syntax.

    Some languages actually let users define their own surface syntax transformations. Macros in Scheme and in Rust do exactly that: they allow transformations to be expressed within the language, and their parser can look up those transformations while parsing, instead of requiring the transformations to be hard-wired into the parser. Same process, slightly more flexible approach. (There's a cost to that approach: the parser needs to be more complex because it needs to look up applicable transformations at suitable points. It might be a very nice project to figure out ways of doing just that.)

    I mentioned that the core of FUNC (our little language with single-argument functions) is essentially the Lambda Calculus.

  • Apr 2:Static Typing

    Slides from the lecture.

    Sample code from the lecture.

    Wikipedia's entry on type systems is reasonable.

    At the end of lecture, I mentioned the problem of giving a reasonable type to reference cells. The sample code types function ref as (:fun (:num) (:ref :num)) which can only create reference cells holding numbers. To create reference cells holding Booleans or any other type of value, you need a distinct function of the appropriate type. A more general type for reference cells uses type variables to give a generic type to ref, such as (:fun [X] (X) (:ref X)). You can create function with such generic types such as the generic identity function 

                    (fun [X] ((a X) X) a)
    The kicker is when calling such generic functions, you need to explicitly supply the types at which you want to instantiate those type variables. For instance: 
                    (let ((id (fun [X] ((a X) X) a)))
                  (id [:num] 10))

                (let ((id (fun [X] ((a X) X) a)))
                  (id [:bool] true))
    Sample code implementing this version of the type system is available. To make sure a type system more palatable (having to specify type instantiation at application time gets old very fast), you can automtically figure out the type instantiations using unification. Languages with these kind of generic types and inference at application time are said to support parametric polymorphism

    Even more advanced type systems can be obtained by using dependent types, which allow types to depend on values! Thus, you can write not only the type of vectors over a type A, but you can also write a type (:vec n A) for vectors of length n over a type A, where n is a value in the language. Using such a type system, we can type a concatenate function on vectors with a function type taking arguments (:vec n A) and (:vec m A) with a return type (:vec (+ m n) A). These type systems have a pretty complex type-checking algorithm.
