Question 1

We can represent a basic SQL query by a Python dictionary of the form:

{ 
  "select": ...,
  "from": ...,
  "where": ... 
}

corresponding to the pieces of a basic SQL query:

SELECT ...
FROM ...
WHERE ...

For example, the representation of the query over the BOOKS and AUTHORED_BY relations from homework 2 (also available in homework3.py) that pulls out the books by Gaiman:

SELECT a.lastName, b.title
FROM BOOKS b, AUTHORED_BY a
WHERE b.isbn = a.isbn AND a.lastName = 'Gaiman'

would be written:

{
  "select": ["a.lastName", "b.title"],
  "from": [ (BOOKS,"b"), (AUTHORED_BY,"a") ],
  "where": [ ("n=n", "b.isbn", "a.isbn"), ("n=v", "a.lastName", "Gaiman") ]
}

The value of the select field of a query is a list of attributes taken from the relations in the from field. Each attribute is qualified, that is, it is prefixed with the nickname of the relation in the query from which the attribute is taken, such as b.title, or a.lastName.

The value of the from field of a query is a list of pairs, each pair (relation , nickname) representing a relation, made up of an instance of a relation and the nickname of the relation that you can use to refer to the relation in the query.

The value of the where field of a query is a list of triples, each triple (op , arg1 , arg2) representing a condition, made up of a string representing the operation of the condition (op), and the first and second arguments to the operation (arg1 and arg2). The overall qualification of the query is the conjunction of all the conditions. An empty list means no conditions. This is of course more limited than SQL, which allows arbitrary Boolean combinations of conditions.

The allowed operations in conditions are the following:

  • n=n : the first and second arguments are qualified attributes, and the condition is true when the values of the qualified attributes are equal.
  • n=v : the first argument is a qualified attribute and the second argument is a value, and the condition is true when the value of the qualified attribute is equal to the second argument value.
  • n>v : the first argument is a qualified attribute and the second argument is a value, and the condition is true when the value of the qualified attribute is greater than the second argument value.

Complete the implementation of function evaluate_query(q) which takes a dictionary representing a query as above, and returns a Relation with the result of the query.

Question 2

We can represent an aggregating SQL query by a Python dictionary of the form:

{ 
  "select-aggr": ...,
  "from": ...,
  "where": ... 
}

corresponding to the pieces of an aggregating basic SQL query:

SELECT ...
FROM ...
WHERE ...

where the SELECT clause aggregates tuples.

The value of the select-aggr field of such a query is a list of triples, each triple of the form ( name , op , attr ) where name is the name you want to give to the result of the aggregation, op is an aggregation operation (one of sum, count, avg, max, min), and attr is the name of a qualified attribute to aggregate with operation op. Thus, the list of triples defines all the components of the resulting tuple, telling you how to create each component.

The value of the from and where fields are as in Question 1.

For example, the query over the BOOKS that sums and averages the pages of all the books:

SELECT SUM(b.numberPages) AS sum_pages, AVG(b.numberPages) AS avg_pages
FROM BOOKS b

would be written:

{
  "select-aggr": [ ("sum_pages", "sum", "b.numberPages"), ("avg_pages", "avg", "b.numberPages") ],
  "from": [ (BOOKS,"b") ],
  "where": [ ]
}

The maximum number of pages of books by Gaiman given by the query:

SELECT MAX(b.numberPages) AS max_pages
FROM BOOKS b, AUTHORED_BY a
WHERE b.isbn = a.isbn AND a.lastName = 'Gaiman'

would be written:

{
  "select-aggr": [ ("max_pages", "max", "b.numberPages") ],
  "from": [ (BOOKS,"b"), (AUTHORED_BY,"a") ],
  "where": [ ("n=n", "b.isbn", "a.isbn"), ("n=v", "a.lastName", "Gaiman") ]
}

Question 3 (for teams of 3 only)

You may wonder why in Questions 1 and 2 we use a different field select and select-aggr. That's because when you want to do aggregation by groups, both are useful!

We can represent a group-aggregating SQL query by a Python dictionary of the form:

{ 
  "select": ...
  "select-aggr": ...,
  "from": ...,
  "where": ... 
  "group-by": ...
}

corresponding to the pieces of an aggregating basic SQL query:

SELECT ...
FROM ...
WHERE ...
GROUP BY ...

where the SELECT clause has both attributes and aggregation attributes.

The value of the select field is as usual a list of attributes, while the value of the select-aggr field is a list of triples containing aggregation information, just like Question 2. The value of the from and where fields are as in Questions 1 and 2. The value of the group-by field is a list of attributes by which to partition the tuples.

As in the last homework (Question 4), the resulting relation will have all the attributes in the select field (which of course must all be attributes listed in group-by) as well as all the aggregates attributes described in select-aggr.

For example, the query over the BOOKS that sums and averages the pages of all the books by each author:

SELECT a.lastName, SUM(b.numberPages) AS sum_pages, AVG(b.numberPages) AS avg_pages
FROM BOOKS b, AUTHORED_BY a
WHERE b.isbn = a.isbn
GROUP BY a.lastName

would be written:

{
  "select": ["a.lastName"],
  "select-aggr": [ ("sum_pages", "sum", "b.numberPages"), ("avg_pages", "avg", "b.numberPages") ],
  "from": [ (BOOKS,"b"), (AUTHORED_BY, "a") ],
  "where": [ ("n=n","b.isbn","a.isbn") ],
  "group-by": ["a.lastName"]
}

Let's spit-and-polish the above. We'll write a small shell for querying a database. This requires some extra code that I've given you in homework3-addendum.py; just append the content of that file to your homework3.py working copy. You will need to install pyparsing.

What's a database? For our purposes, a database is a dictionary from names to relations. A name is just a string by which we can refer to a table in a query, and a relation is an instance of the Relation class above.

So a simple database for our sample relations might be:

sample_db = { 
  "Books": BOOKS,
  "AuthoredBy": AUTHORED_BY,
  "Persons": PERSONS
}

Mmm. Okay. How does that do anything? Well, I've provided you with a parser (a function parseQuery) that takes a string containing a basic SQL query, and translates it into what I'll call an abstract query of the form:

{ 
  "select": ...,
  "from": ...,
  "where": ... 
}

which is just like a query from Question 1, except that the value of the from field an abstract query is a list of pairs, each pair (name , nickname) representing a relation, made up of a string containing the name of a relation in the database and the nickname of the relation that you use to refer to the relation in the query. Thus, the basic SQL query:

SELECT a.lastName, b.title
FROM Books b, AuthoredBy a
WHERE b.isbn = a.isbn AND a.lastName = 'Gaiman'

can be represented (and in fact, will parse as) the abstract query:

{
  "select": ["a.lastName", "b.title"],
  "from": [ ("Books","b"), ("AuthoredBy","a") ],
  "where": [ ("n=n", "b.isbn", "a.isbn"), ("n=v", "a.lastName", "Gaiman") ]
}

Note the differences between this and the kind of query in Question 1.
