Databases (Fall 2021)

This is an introduction to database systems. Topics covered include: the relational model, SQL-based querying, transactions, data structures, as well non-relational database models. The approach will be hands-on: while we will practice on existing database systems, we will also write code to implement simplified versions of the various database models and their query systems.

  • Sep 27: The Relational Model

    Reading: Textbook, Chapter 6.

    Wikipedia on The Relational Model.

    The original paper of Codd on the relational model, which is of historic interest. Also interesting is Codd's 1981 Turing Award lecture.

    SQLite is a small relational database engine that stores data in a single file and can be accessed programmatically from many programming languages. It's useful in its own right, but it's great for learning about the basics of relational data. It is also generally installed by default on Linux and Mac OS X installations. Try typing sqlite3 from a Terminal.

  • Oct 4: Relational Databases and SQL

    Reading: Textbook, Chapter 7.

    Wikipedia page on SQL.

    A draft of the 1992 standard, if you like reading those sort of documents.

    There are tons of tutorials on SQL online, usually tied to a particular database implementation, since you can't use SQL "in the abstract". I've used Postgres's SQL tutorial in the past to introduce SQL to colleagues (start with Section 2). An interactive tutorial that I greatly enjoy is SQLBolt.

    The sample SQLite interactions I demoed in class.

    Two SQL features we did not cover in class but you may run into are views (which are a form of live query that looks like a table in the database) and stored procedures (which are a form of functions that can be written directly into the database). Views tend to look the same across relational databases, while stored procedures are much more vendor dependent.

  • Oct 18: Normalization and Indexing

    Reading: Textbook, Chapter 6.2 (normalization) and Chapter 12 (physical storage and indexing)

    A reasonable set of slides on normalizations and functional dependencies. It goes into a bit more details about decomposition, including the notion of lossless-ness.

    Overviews of physical storage and indexing tend to be hefty because there are many axes across which to compare the various approaches. A reasonable writeup (which provides a compatible alternative to the textbook) is this set of lecture notes.

    Wikipedia pages on B trees and B+ trees, data structures for managing indexes on block-based storage systems (like disks).

  • Oct 25: Transactions

    Reading: Textbook, Chapter 14.

    Relevant Wikipedia entries: Transactions, Serializability, Two-phase locking, ACID properties.

  • Nov 1: Programming APIs and ORMs

    Reading: Textbook, Chapter 15.

    JDBC is the Java API for connecting to databases. Most databases have JDBC drivers that support that API. It has become a de facto standard, and many other languages use that same API to provide database connectivity.

    ORMs (Object-Relational Mappings) are a more abstract way to provide an interface to a database through a programming language: ORMs reify database tables as data structures in the language, so that modifying the structures will automatically update the underlying tables, and changes to the underlying tables will be visible through the structures.

    LINQ is one approach to integrate SQL-style querying within a programming language (here, C#).

    Object persistence is the idea that we can create objects in a programming language and have those objects automatically be made persistent by transparently storing them in a database. Java does not do this automatically, but serialization is the main mechanism to achieve persistence. JADE is an example of a programming language that supports object persistence natively and transparently.

    Transactional memory is an approach to programming concurrent systems in a way that does not require locks, by essentially moving the notion of transactions to the world of reading and writing memory within a single machine.

  • Nov 8: Distribution and Replication

    Reading: Textbook, Chapter 16.

    Distributed databases are distributed into nodes either by replication or by partitioning (or a combination of both).

    The two-phase commit protocol, which ensures that database nodes agree on whether a distributed transaction should commit or abort.

    The CAP Theorem.

    Lamport's Paxos protocol (well, protocol family) is one of the most common ways to achieve consensus on what the data stored in a distributed system is when some of the nodes disagree due to update messages not getting to them. It is worth understanding. Here is a two-parter that does a reasonable job explaining the details: Part 1, Part 2.

    A research survey on The State of the Art in Distributed Query Processing. Reasonably readable if you're willing to spend some time on it.

    A good introduction to distributed systems with an emphasis on databases, and which covers not only what I talked about for this lecture, but also hints at some of the things we'll cover when talking about NoSQL databases.

  • Nov 15: NoSQL Databases

    Reading: Textbook, Chapter 11.

    An old (2011) survey of NoSQL Databases by Strauch. It gives a good overview of the field.

    BASE: An ACID Alternative by Pritchett.

    Redis is one of the most common key-value stores. Amazon's Dynamo paper which describes one of the first high-availability key-value store. There is a nicer primer for it as well.

    Wikipedia entries on consistent hashing and distributed hash tables, the main technique for sharding in NoSQL systems.

    A good overview of document-oriented data stores and their advantages. MongoDB is a popular document-oriented data store. It supports transactions and sharding.

    Google's Spanner database is an interesting example of a system that started out as a key-value store and that evolved into a distributed relational database system. A great technical overview is an OSDI 2012 paper.

  • Nov 29: Graph Databases

    Old lecture slides from Stanford describing typical graph representations and algorithms.

    Neo4j is the prototypical graph database. Here are two excerpts from the book Graph Databases, which is mostly about Neo4j: the query language, and the internal representation. You might be able to download the book for free.

    Cypher is Neo4j's query language. The docs have an interesting complete example of using Cypher to drive a simple recommendation engine. There's also a useful comparison with SQL.

    FlockDB was Twitter's graph database engine for storing user relational data such as followings and favorites. Its source code is available. (Interestingly, it is written in Scala.)

    TAO is used at Facebook (at least as of a few years ago) to store and serve the social graph. A technical paper describing TAO appears in the Usenix 2013 Technical Conference.

    Although the name suggests it is a graph-based query language, GraphQL is not related to graph databases. It is, quoting from the working draft of the specification, "a query language and execution engine originally created at Facebook in 2012 for describing the capabilities and requirements of data models for client-server applications". Still worth knowing though, since it's becoming increasing popular as an API query language.
